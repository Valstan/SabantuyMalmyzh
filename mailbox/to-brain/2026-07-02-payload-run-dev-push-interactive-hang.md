# Грабля: `payload run <script>` беззвучно виснет на интерактивном dev-push

## Симптом

`corepack pnpm -C web payload run src/seed/x.ts` на dev (push:true) **висит без
единого байта вывода** (node жрёт CPU и замирает). Никаких ошибок, никаких
промптов в захваченном stdout. Легко принять за БД-лок или зависший pnpm.

## Причина

Payload на init в dev запускает drizzle-push. Если схема-дифф **деструктивный**
(в конфиге текущей ветки НЕТ коллекции, чьи таблицы уже есть в dev-БД — типовой
случай: переключился на ветку без новой коллекции, а таблицы в общей dev-БД
остались), drizzle задаёт **интерактивный вопрос про DROP** и ждёт stdin.
В неинтерактивном запуске (агент, CI, фон, pipe) ответа не будет никогда →
вечное молчаливое зависание. Аддитивный дифф проходит без вопросов — поэтому
грабля выстреливает редко и внезапно.

## Лечение / правило

Сиды и разовые скрипты, которым push не нужен, гонять с **`NODE_ENV=production`**
(push выключен, как на проде):

```bash
NODE_ENV=production DATABASE_URL=… PAYLOAD_SECRET=… corepack pnpm payload run src/seed/x.ts
```

Диагностический рефлекс: «payload-процесс висит, 0 байт вывода» → первым делом
подозревать интерактивный push-промпт (расхождение ветка↔dev-БД), а не локи БД.

## Переносимость

Любой Payload 3 + postgres-адаптер с `push:true` в dev и общей dev-БД на
несколько веток/чекаутов — GONBA ровно в той же конфигурации.
