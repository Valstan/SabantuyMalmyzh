## Грабля

Переключение git-ветки (checkout/merge/branch -b) при **работающем** `next dev`
(Next 15, webpack) травит module-граф: в момент переключения файл новой ветки
на миг отсутствует на диске, webpack кэширует «Module not found / Failed to
read source code … (os error 2)» — и **не восстанавливается**, даже когда файл
уже на месте и позже редактируется (HMR подхватывает правки соседних файлов,
но отравленный узел остаётся мёртвым).

## Симптомы (все словил за одну сессию, трижды)

- Страница целиком: 500 или «Application error: a client-side exception…».
- Коварнее: **компонент молча отсутствует** (клиентский остров не гидрируется,
  кнопка есть, а состояние не реагирует) — консоль сервера при этом держит
  старую ошибку «Failed to read source code», браузерная может быть чистой.
- Вдвойне коварно с service worker: SW отдал закэшированные ЧАНКИ удалённого
  `.next` → hydration-крах уже от другой причины (маскирует первую).

## Лечение / профилактика

- Лечение: **рестарт dev-сервера**; при упорстве — `rm -rf .next` перед стартом;
  если на сайте SW — почистить и клиентские кэши/регистрацию
  (`caches.delete` + `unregister`), иначе старые чанки переживут рестарт.
- Профилактика: **останавливать dev-сервер перед git checkout / merge / branch**.
  Дёшево и снимает весь класс.

## Почему в pool

Переносимо на весь Payload/Next-кластер (GONBA — тот же стек и тот же
PR-flow с частыми переключениями веток). Диагноз неочевиден: симптом
(«компонент не работает») выглядит как баг свежего кода, а причина — кэш
compile-графа от момента переключения ветки. Маркер для GOTCHAS: grep
серверного лога на `Failed to read source code` → это ОНО, рестарт, не дебаг.
