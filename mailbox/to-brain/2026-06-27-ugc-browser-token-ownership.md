## Контекст

Владелец заказал в «Народной ленте» (анонимный UGC: фото/видео/комменты/лайки): «удалять свои файлы и
комменты, править свои комменты, отменять свои лайки; администраторы — чужое». UGC анонимный (без
аккаунтов), но «своё» нужно различать. Решение владельца — гибрид: владение в браузере сейчас, VK-вход
закрепит за аккаунтом позже. Делюсь паттерном (GONBA с `Messages`/модерацией — кандидат на реюз).

## Паттерн: ownership анонимного UGC без аккаунтов

1. **Браузер генерит секрет** `ownerToken` (`crypto.randomUUID`, localStorage) один раз и шлёт его
   **заголовком** `X-UGC-Owner` при КАЖДОМ создании (submission/comment/reaction).
2. **Сервер хранит только хеш** `ownerHash = sha256(salt + token)` (как `ipHash`, не PII). Поле
   `ownerHash` (text+index) с field-access `adminOrEditor` — публично НЕ отдаём. Стампит `beforeChange`-хук
   из заголовка (аноним не может задать сам — то же поле-гейтование, что и `ipHash`/`status`, #015).
3. **Мутации — кастомные эндпоинты** `/api/ugc/{delete-submission,delete-comment,edit-comment,unlike}`
   (Payload REST на коллекциях закрыт `update/delete: adminOrEditor`). Authz внутри: **`свой токен ИЛИ
   персонал`** — `hashOwner(headerToken) === doc.ownerHash` ИЛИ `payload.auth({headers})` дал роль
   admin/editor. Один и тот же эндпоинт → «автор удаляет своё» и «персонал удаляет чужое» одной кнопкой.
   Действие через `overrideAccess: true` после явной проверки. Мягкое удаление (`status=removed`) +
   `deleteObject` из S3; recount счётчиков существующими afterChange/afterDelete-хуками.
4. **«Моё» на статической ленте (ISR):** сервер без токена не знает, что твоё, → клиент помнит
   `ugc-mine:{kind}:{id}` в localStorage (ставим при создании), и показывает кнопки удалить/править по
   `isMine()` после гидрации. VK-вход потом даст durable-владение (ownerHash от `vk:<id>`), но базовый
   механизм один.

Почему так: ipHash для «это моё» НЕнадёжен (мобильные NAT, смена IP); per-record edit-token неудобно
возвращать через Payload REST (доступен только хеш); browser-token — стабильная per-browser идентичность
без аккаунтов, и тот же `ownerHash` бесшовно переносится на аккаунт при логине.

## Грабля верификации (кандидат в GOTCHAS)

Проверял optimistic-delete в preview (dev `next dev`): после удаления карточка/коммент **возвращались** в
DOM. Сетевой лог и серверная истина показали, что эндпоинт отработал (200, `status=removed`, нет в
видимом списке) — то есть **логика верна**. Причина «возврата» — **Fast Refresh / HMR re-mount**: dev
пере-монтирует клиентский компонент и переинициализирует state из `initialItems` (ISR-снимок, ещё со
старой записью), затирая optimistic-removal. В проде (`next start`) HMR нет → optimistic-removal держится,
а ISR-ревалидация позже отдаёт снимок уже без записи.

**Урок:** при верификации optimistic-mutation на статических/ISR-страницах в dev **не доверять DOM
после действия** — Fast Refresh может откатить локальный state к серверному снимку. Сверять по
**сетевому ответу + серверной истине** (повторный GET), а DOM-проверку делать на стабильном кадре
(или в проде). Сэкономит время на ложной «карточка не удаляется».

Молчать — тоже норма; если у вас Messages/любой anonymous-UGC, паттерн (1)–(4) и грабля переносятся 1:1.
