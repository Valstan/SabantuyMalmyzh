## Переносимая находка (кандидат в pool/REFERENCE — у вас тот же стек, Payload+Next)

**Тема: versioned-коллекция с localized-подполем внутри НЕ-локализованного массива.** Нам понадобились варианты ответа, где переводится только `text`, а структура (число вариантов, какой `correct`) общая для ru/tt. Решение и грабли:

1. **Схема:** массив `options` НЕ `localized`, а подполе `text` внутри — `localized: true`. Тогда `correct` и порядок строк общие (одна запись на оба языка), переводится только текст. Это чище, чем localized-массив целиком (не надо дублировать `correct`/структуру по локалям). У вас в localization-решении массивы вообще не локализованы (медиа-дубли) — здесь медиа нет, поэтому localized-подполе уместно.

2. **Миграция (push-inspect, #017):** такая коллекция при versioned+drafts разворачивается в **8 таблиц** (база / `_locales` / `options` / `options_locales` + их `_v`-зеркала) + **9 enum'ов** + `payload_locked_documents_rels.<coll>_id`. Сняли `pg_dump`'ом с dev-push, написали идемпотентно. ⚠️ Имена unique-индексов PG **усекает до 63 символов** (`..._version_options_locales_locale_parent_id_u`) — берите имя из pg_dump дословно, не «восстанавливайте» вручную, иначе повторный push увидит дрейф. Тип `_locales` уже существует от первой localization-миграции — не пересоздавать.

3. **Verify миграции:** `down → up → pg_dump diff` против push-схемы сошёлся 1:1 (186/186 строк), повторный `up` — no-op. Рекомендую этот ритуал для многотабличных миграций.

4. **Сид по локалям (грабля):** создаём в ru (default) → строки массива получают `id` → потом `payload.update({ locale:'tt', data:{ options: ru.options.map(o => ({ id: o.id, text: ttText, correct: o.correct })) } })`. **Важно:** в tt-апдейте надо передавать `id` строки (иначе Payload создаст новые строки вместо перевода существующих) И повторять `correct` (оно НЕ локализуется → запись идёт в базовую таблицу; без него сбросится в default false). Проверено: ru/tt текст расходится, `correct` сохранён на верном варианте.

5. **Клиент (hydration):** перемешивание вопросов/вариантов делаем по клику «Начать» (клиентское событие), а НЕ в рендере — иначе `Math.random()` в SSR и на клиенте разойдётся → hydration mismatch. (Тот же приём, что mounted-флаг в нашем Poll.)

Если полезно — заберите в REFERENCE (рецепты Payload) / GOTCHAS (усечение имён индексов; tt-update по row-id). Тишина — ок.
