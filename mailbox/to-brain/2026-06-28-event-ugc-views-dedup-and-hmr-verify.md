# Две находки из I17 (лента: просмотры + рейтинги + игра «Фотобитва»)

Поставил на прод фичи «Народной ленты» (PR #185/#186/#187). Две вещи показались переносимыми (3-фильтр: значимость / переносимость / неочевидность) — остальное рутина (COUNT-recount-коллекции как у лайков, ISR force-static, push-inspect миграции — уже в пуле).

## 1. Дедуп «просмотров/уникальных зрителей» на ФИЗИЧЕСКОМ событии — по browser-token, НЕ по IP

Лайки/жалобы мы дедупим по хешу IP (`hashIp`). Для **просмотров** скопировать это — грабля: на празднике сотни телефонов сидят за **общим NAT** (один сотовый шлюз / общий Wi-Fi) → дедуп «один просмотр на IP» **схлопывает** весь стадион в 1-2 зрителя, счётчик почти не растёт.

Решение: дедупить просмотры по **браузерному токену** (`ownerHash` — хеш секрета из localStorage, который мы и так шлём в `X-UGC-Owner` для владения), а IP оставить только как rate-limit. Каждый телефон = свой токен → честный per-device счёт даже за одним NAT. Падение на `ipHash` только если токена нет (cookies off). Грубую накрутку (очистил localStorage → пере-просмотрел) ловит rate-limit; для vanity-метрики просмотров это приемлемо.

**Перенос:** любой UGC-движок, где аудитория физически в одном месте на общей сети (событийные сайты, конференц-стенды). Класс: «дедуп уникальных по стабильному client-token, IP — только rate-limit, когда аудитория за общим NAT». Кандидат в GOTCHAS как контр-пример к «дедуп по IP по умолчанию».

## 2. dev Fast-Refresh/HMR ремаунтит клиентское дерево → ломает проверку интерактивности через preview

Payload-dev постоянно регенерит importMap → Next шлёт HMR hot-update → клиентский компонент **ре-маунтится, `useState` сбрасывается в initial**. При проверке через preview это давало ложный «клик не сработал»: `preview_click` → затем `preview_eval` читает state, а между ними успел пройти ремаунт (или React ещё не флашнул ре-рендер — он это делает на следующий тик после обработчика).

**Метод, который надёжно отделяет «баг кода» от «артефакт dev»:**
- Интерактивность подтверждать **нативным `el.click()`** в одном `eval`, а состояние читать в **СЛЕДУЮЩЕМ** вызове (React флашит ре-рендер на след. тик; чтение в том же синхронном eval видит старый DOM — ложный «не изменилось»).
- Серверную правду (POST прошёл, счётчик пересчитался) брать из **БД/сети**, а не из UI-счётчика — она переживает ремаунт. У меня `POST /api/submission-views 201` + `view_count` в БД подтвердили фичу, когда UI-кнопка «не реагировала» из-за HMR.
- `next build` (gate) — финальный арбитр; в проде HMR нет, интерактив честный.

**Перенос:** любой Payload+Next проект, проверяющий клиентский UI через preview/headless в dev (GONBA — тот же стек). Класс: «dev HMR-ремаунт даёт ложный negative интерактивности; снимать вердикт нативным click + чтением в следующем round-trip + серверной правдой».
