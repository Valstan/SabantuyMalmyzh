# LiveInternet видимый счётчик: первый подозреваемый «не видно» — клиентская блокировка, не код

## Контекст

Владелец сообщил: видимый бейдж LiveInternet (тип `t29.6`, 88×120, в подвале) не виден
на сайте, Ctrl+F5 не помогает. Счётчик ранее выкатили в PR #115. Разбор занял сессию —
делюсь, потому что причина оказалась неинтуитивной, а стек/аудитория совпадают с GONBA
(Payload+Next, рунетовский контент-сайт).

## Что подтвердилось фактами

1. **Разметка бейджа была на проде** (греп HTML: `li-counter`, `licnt…` присутствуют),
   CSS не прятал, build-time флаг `NEXT_PUBLIC_LIVEINTERNET=1` был.
2. **Трекинг работал** — на `liveinternet.ru/stat/<домен>/` живые посетители за сегодня.
   Значит хит долетал, счётчик зарегистрирован, статистика открыта.
3. Серверный `fetch` на `counter.yadro.ru/hit?t29.6;…` **всегда** возвращал 43-байтный
   прозрачный GIF 1×1 — при ЛЮБОЙ форме домена (punycode/кириллица/www/http) и ЛЮБОЙ
   кодировке (`escape`/`encodeURIComponent`). ⚠️ **Вывод-ловушка:** по этому легко
   ошибочно заключить «тип счётчика невидимый / бейджа нет». На деле `/hit` **отдаёт
   видимую картинку только живому браузеру в контексте страницы на боевом домене** —
   серверный fetch её не воспроизводит в принципе. Байт-сайз ответа `/hit` — **НЕ**
   валидный тест видимости бейджа.
4. **Подтверждённая причина «не видно»:** в браузере владельца **анти-баннер** (рунетовский
   ad/tracker-блокировщик — Kaspersky «Анти-баннер» / AdGuard / защита от слежки) **резал
   `counter.yadro.ru`**. Отключил анти-баннер → бейдж сразу стал виден.

## Переносимые уроки

- **`counter.yadro.ru` (LiveInternet) — в чёрных списках многих рунетовских блокировщиков
  и антивирусов.** Следствие для любого сайта с видимым LI-счётчиком: (а) часть реальных
  посетителей не увидит бейдж и не будет посчитана LI; (б) **источник правды по трафику —
  Яндекс.Метрика** (блокируется реже), LI — скорее «рейтинг/витрина», не точная аналитика.
  При диагностике «бейдж не виден» **первый подозреваемый — клиентская блокировка, не код**
  (проверка: F12 → Network → фильтр `yadro` → запрос `blocked`/`failed`? → инкогнито без
  расширений).
- **Не «улучшать» готовые сниппеты вендора-аналитики — воспроизводить 1:1, гейтить env'ом.**
  В PR #115 у нас `escape(...)` (как в официальном коде LI) был отрефакторен в
  `encodeURIComponent(...)`. Откатили к `escape` ради точного соответствия сниппету из
  кабинета владельца. Гипотеза (НЕ подтверждена строго — мешал тот же анти-баннер): для
  IDN-домена `.рф` `document.URL` в браузере приходит кириллицей, `escape` кодирует её в
  `%uXXXX` (формат, который бэкенд LI ждёт для сопоставления домена), а `encodeURIComponent`
  — в UTF-8 `%XX`; рассинхрон мог ломать привязку домена → пиксель вместо бейджа. Доказать
  не вышло (нельзя изолировать от анти-баннера), но вывод-практика твёрдый: **canonical
  vendor snippet = воспроизводить дословно, escape() там load-bearing.**

## 3-фильтр

Значимо (видимый счётчик — частый owner-запрос; молчаливый недосчёт LI вводит в
заблуждение), переносимо (GONBA — тот же стек + рунетовская аудитория, может ставить
счётчики), неочевидно (counter.yadro.ru-в-блоклистах и escape-vs-encodeURIComponent для
IDN — обе грабли непонятны до удара). Кандидат в `cross-project-ideas/GOTCHAS.md`.
