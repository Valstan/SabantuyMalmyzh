# Грабля: data-driven sitemap пререндерится в сборке против пустой CI-БД → молча вырожден на проде

Делюсь по [#009](../../cross-project-ideas/ideas/009-share-findings-reflex.md). **GONBA — тот же стек (Payload+Next standalone, сборка в CI с эфемерным postgres, ISR) → почти наверняка имеет тот же латентный баг.** Кандидат в [GOTCHAS](../../cross-project-ideas/GOTCHAS.md). Compliance=recommend именно из-за прямой применимости к GONBA.

## Симптом

Прод `/sitemap.xml` отдавал **только статические записи** (`/`, `/map`, `/gallery`) — события/страницы/альбомы отсутствовали. При этом:
- главная (`getPayload` + events, `revalidate=60`) на проде рендерит события нормально;
- схема/данные на проде здоровы, прямой SQL отдаёт published-доки;
- **локально** sitemap полный;
- в прод-логах **никаких ошибок** (даже после того, как обернул каждый запрос в try/catch с `console.error` — лог пуст).

## Корень (не то, на что грешишь сначала)

`app/sitemap.ts` с `export const revalidate = 3600` — это **статически пререндеримый** route. Next генерит его **на этапе `next build` в CI**, а CI собирает против **эфемерной пустой build-БД** (`sabantuy_build`, push:true создаёт схему, контента нет). Результат запекается в бандл: `.next/server/app/sitemap.xml.body` (+`.meta`) — вырожденный sitemap с одними статическими маршрутами. В рантайме Next **отдаёт этот пререндер** и считает его свежим весь `revalidate`-период (3600с) → `getPayload` в рантайме **не вызывается** (потому и логов нет). После каждого деплоя sitemap вырожден на час; деплоим часто → практически всегда вырожден.

Почему главная не страдала: у неё `revalidate=60` → первый рантайм-хит через минуту триггерит фоновую регенерацию против РЕАЛЬНОЙ прод-БД. Sitemap с 3600 «лечился» бы только через час — а до него не доживал (следующий деплой запекал пустой заново).

Диагностический маркер: в задеплоенном бандле лежит `*.body`/`*.meta` для route → значит он пререндерен в сборке (с тем контентом, что был в build-БД), а не строится в рантайме.

## Лечение

```ts
// app/sitemap.ts
export const dynamic = 'force-dynamic'   // вместо revalidate=<N>
```
Убирает build-пререндер → sitemap строится в рантайме против реальной прод-БД на каждый запрос. Запросы лёгкие (`depth:0`), краулеры ходят редко → дёшево. После фикса прод-sitemap: 3 → 28 URL (события+страницы+альбомы появились).

Альтернатива при желании кэшировать: оставить ISR, но не пускать пустую БД в build-пререндер (напр. `generateSitemaps`/гард, либо короткий `revalidate` как у остальных страниц). Для sitemap force-dynamic проще и корректнее.

## Эвристика

> Любой **data-driven** static/ISR route (sitemap, RSS, OG-листинги), который строится `next build`'ом, в CI берёт данные из **эфемерной build-БД** — если она пустая, контент запекается пустым и живёт весь ISR-период на проде. Симптом обманчив: «локально полно, на проде пусто, ошибок нет». Для таких routes — `force-dynamic` (рантайм-данные), либо наполнять build-БД, либо короткий revalidate. Не путать с трейсингом/`getPayload` (это была ложная версия — getPayload в route работает).

## 3-фильтр

- **значимость** — молча деградирует SEO-sitemap на проде, без единой ошибки в логах → можно не заметить месяцами;
- **переносимость** — любой Payload+Next+CI-build-with-ephemeral-DB; **GONBA напрямую** (проверьте их `/sitemap.xml` на проде: есть ли там реальные доки или только статика);
- **неочевидность** — выглядит как баг запроса/трейсинга; на деле кэш build-пререндера.
