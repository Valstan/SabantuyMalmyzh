# ISR кэширует пустой рендер при транзиентном сбое запроса к БД

## Симптом

На проде главная (`/`, ISR `revalidate=60`) **иногда показывает пустое расписание**
(«Расписание пока не опубликовано»), хотя события в БД есть (API отдаёт 12). Через
~минуту само возвращается. Поймал вживую через Chrome-расширение: один заход —
пусто, перезагрузка с новым URL — все события на месте.

## Механизм

Наши data-хелперы в server-компоненте обёрнуты в `try/catch`, возвращающий `null`
на ошибке (паттерн «секция не валит всю страницу»):

```ts
async function getPublishedEvents(locale) {
  try {
    const payload = await getPayload({ config })
    const res = await payload.find({ collection: 'events', /* … */ })
    return res.docs
  } catch {
    return null            // ← на транзиентном сбое БД отдаём null
  }
}
```

Когда **ISR-ревалидация** совпала с моментальным сбоем `payload.find` (коннект-пул
моргнул / БД на миг занята), хелпер вернул `null` → `items=[]` → отрендерился
placeholder «пусто». **Next закэшировал этот пустой рендер** и отдавал его всем до
следующей ревалидации (≥60с, а при низком трафике — дольше, пока кто-то не дёрнет
страницу после истечения revalidate). То есть единичный транзиентный сбой
«отравляет» кэш на всё окно, и это **молча самовыправляется** — оттого почти
незаметно в проде.

## Почему пишу (3-фильтр)

- **Значимость:** пользователь видит «пустой» сайт (расписание/галерея/любая
  data-driven секция) после случайного сбоя БД, хотя данные есть.
- **Переносимость:** GONBA — тот же стек (Next 15 ISR + Payload + Postgres) и тот же
  паттерн `try/catch → null` в server-хелперах почти наверняка есть → задета так же.
- **Неочевидность:** `try/catch→null` выглядит как «безопасная деградация», но в
  связке с ISR превращается в «закэшировать пустоту». Самохилинг маскирует баг.

## Возможные защиты (на обсуждение)

1. **Ретрай** запроса 1–2 раза с маленькой паузой перед тем, как отдать `null`
   (большинство транзиентных сбоев пула лечатся повтором).
2. **Не кэшировать сбойный рендер:** если критичная секция пуста ИЗ-ЗА ошибки (а не
   потому что данных правда нет) — пробросить ошибку, чтобы ISR не зафиксировал
   пустой результат (Next отдаст прошлый хороший кэш / 500 → не закэшируется). Тонко:
   надо отличать «пусто по данным» от «пусто из-за сбоя» (флаг из catch).
3. Снизить риск: единый `getPayload` инстанс/ретрай-обёртка на уровне data-слоя.

У нас severity низкая (прод почти пуст, трафика нет, само выправляется) — занесли в
PENDING как инженерный пункт «при касании HomeView». Но для GONBA с реальным
трафиком и контентом это может бить заметнее — потому шлю в pool.
