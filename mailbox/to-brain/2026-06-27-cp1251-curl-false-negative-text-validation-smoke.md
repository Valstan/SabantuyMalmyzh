# Находка (suggest, low): cp1251-мангл инлайн-curl → ложно-зелёный смоук валидации текста

## Контекст

PR2 «Народной ленты»: коллекция `submissions` с хуком-санитайзом, в т.ч. простой
стоп-фильтр мата по кириллическим корням. Локальный API-смоук на живом dev:
`POST /api/submissions` с `caption:"иди на хуй"` — **ожидал 400, получил 201**.

## Ловушка (почему дорогая)

Это выглядело как **баг кода** (фильтр не ловит мат) и увело в отладку не туда:
проверял regex-lookbehind, флаг `u`, бандл Next/SWC, перезапуск payload-синглтона,
сносил `.next` — несколько раундов. Изолированный node-тест фильтра при этом
проходил. Decisive-инструментация (лог `data.caption` в хуке) показала: в хук
приходит **`"??? ?? ???"`** — кириллица побита.

Причина — **семья G11**: инлайн-`curl -d "{...кириллица...}"` в Windows-консоли
(Git Bash) кодирует кириллицу в **cp1251**, curl шлёт cp1251-байты, сервер декодит
тело JSON как UTF-8 → мождибейк в строковом поле. Фильтр видел мусор → пропускал.

**Коварство:** смоук был **ложно-ЗЕЛЁНЫМ** в опасную сторону. 201 читается как
«валидатор корректно пропустил чистый текст», а на деле он пропустил **мат**. Не
крэш, не явный 500 — тихий false-negative ровно на проверке безопасности контента.

## Лечение (переносимо)

Любой смоук, где **в теле запроса есть не-ASCII и его проверяет сервер**
(текст-валидация, стоп-фильтры, поиск, нормализация, i18n-поля):

- тело писать в **файл в UTF-8** (любой редактор/`Write`, не консольный heredoc) и
  слать `curl --data-binary @body.json -H 'Content-Type: application/json; charset=utf-8'`;
- **не** полагаться на инлайн-`-d "{...кириллица...}"` в Windows-консоли;
- при «странном» результате текст-смоука — **сначала проверить кодировку тела**
  (залогировать, что реально пришло на сервер), потом лезть в код.

Рядом с G11. Кандидат — самостоятельная грабля «cp1251 инлайн-curl → ложно-зелёный
смоук текст-валидации» (методология теста, не код). Переносимо на любой
RU/мультиязычный проект, который смоучит API текст-валидацию с Windows-машины.
