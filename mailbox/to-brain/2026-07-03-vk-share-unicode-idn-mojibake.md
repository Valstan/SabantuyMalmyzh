# Грабля: репост в ВК с кириллическим доменом → «?4??4??4?…» вместо ссылки

**Симптом.** Владелец делится страницей сайта в сообщество ВК — в опубликованном
посте вместо ссылки видна каша `?4??4??4??4?….?4??4?….?4??6?` (по мусор-токену на
каждую букву домена). Кликабельность сохраняется, но выглядит сломано.

**Причина (доказано на живом посте + e2e).** В пост ушёл URL с **юникод-хостом**
(`https://сабантуй.вмалмыже.рф/…` — вытащен из `away.php?to=` href испорченного
поста). ВК при публикации запекает юникод-домен в текст сниппета уже испорченным.
Тот же URL в **punycode** (`https://xn--…/…`) ВК рендерит идеально: домен
показывается человеческой кириллицей «сабантуй.вмалмыже.рф» (проверено сквозняком
share.php → превью → публикация в ЛС «Избранное»).

**Откуда берётся юникод-URL, если в коде везде punycode:** пользователь делится
НЕ кнопкой сайта — системное меню телефона / «поделиться страницей» из браузера /
копия из адресной строки: мобильные браузеры отдают display-форму IDN (юникод).

**Лечение (перенос 1:1 на любой сайт с .рф-доменом, GONBA — тот же случай):**
1. Хелпер `asciiUrl(u) = new URL(u).href` — URL API сам каноникализирует хост в
   punycode (и percent-encodes юникод в пути/квери). Прогонять через него **все**
   URL, уходящие наружу: соцссылки (share.php/t.me/connect.ok.ru), `navigator.share`,
   копирование в буфер, и особенно fallback `window.location.href`.
2. Плюс per-page OG (`og:url`/canonical на punycode-домене) — скрейперы соцсетей
   берут их при разборе ссылки.

Диагностический рефлекс: мусор `?N??N?…` с точками, число токенов = число букв
домена → это юникод-IDN, а не кодировка страницы.
