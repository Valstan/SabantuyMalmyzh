# Находка для библиотеки: `curl` + не-ASCII body на Windows/git-bash → тихий mojibake

## Симптом
При разработке M2 (форма регистрации) Payload-коллекции получали **пустой авто-slug** и **mojibake-заголовки**, хотя `slugField` (`beforeValidate` + `formatSlug` с `\p{L}`/`\p{N}`, флаг `u`) был полностью корректен.

## Корень
Тест-харнесс слал заявки через `curl -d '{"title":"Кириллица"...}'` из git-bash на cp1251-Windows. Тело уходило **не в UTF-8**, сервер принимал битые байты. Для slug это особенно коварно: мусорные байты — не `\p{L}` → регексп `[^\p{L}\p{N}-]` их вырезает → `formatSlug` возвращает `""` → slug пустой. Заголовки же сохранялись как mojibake.

Диагностика (что отсекло код от харнесса):
- юникод-self-test в хуке: `'абв'.replace(/[^\p{L}]/gu,'X')` → `'абв'` (литерал из исходника — ок, `u`-флаг работает);
- но `computed=""` для значения **из запроса** → значит битый именно вход, а не регексп;
- пересоздание того же события через **node fetch** (`JSON.stringify` = UTF-8) → slug `"праздничный-концерт"` сразу корректный.

## Рекомендация (для GOTCHAS / REFERENCE)
- На Windows/git-bash для API-тестов с кириллицей/не-ASCII использовать **node fetch**, а не `curl -d`. Либо писать тело в файл как UTF-8 через node и слать `curl --data-binary @file`.
- Эвристика отладки: «пустые/битые не-ASCII значения после записи через CLI» → **сперва проверь кодировку харнесса** (повтори через node fetch), потом ищи баг в коде. Экономит час+.
- Переносимо на любой проект, тестирующий HTTP API из git-bash на Windows (в т.ч. GONBA — тот же стек/ОС).

Связано: возможно, новая запись в `cross-project-ideas/GOTCHAS.md` (симптом: пустой slug / mojibake при записи через curl).
